Uso de Formularios

Los formularios en HTML se utilizan para recopilar datos de los usaurios. Proporcionan una forma para que los usuarios introduzcan información como texto, contraseña y señecciones, y luego envíen esos datos a un servidor para su procesamiento. Los formularios se creen utilizado el elemento, que contiene varios elementos de entrada como y para definir los diferentes campos de entrada del usuario.

¿Qué es <label> y para qué sirve?

<label> es el texto/caption que describe un control de formulario (como un input)

✅ ¿Por qué es tan importante?

Asociar correctamente un <label> con su contro trae ventajas claras:

  • Accesibilidad: el lector de pantalla lee el label cuando el usaurio cea en el input, ayudando a entender qué datos se debe ingresar.
  • Usabulidad: hacer clic/tocar el texto del label mueve el foco al input (y en chackbox/radio lo activa), aumentando el "área clickeable", útl en móvil.

🔗 Formas de asociar <label> con <input>

Hay dos formas correctas: asociación explícita (recomendada) e implícita.

A) Asociación explícita (recomendada): for + id

El for del <label> debe coincidir exactamente con el id del <input>. 

        
          <label for="email">Correo</label>
          <input id="email" name="email" type="email" autocomplete="email" required>
  • for="email" ⟷ id="email" (match exacto
  • El label queda programáticamente asociado y es clickeable.

📌 Nota: un <label> se asocia a un solo control, pero un control puede tener múltiples labels apuntando al mismo id.

B) Asociación implícita: anidar el <input /> dentro del <label>

Si metes el input adentro del label, no necesitas for/id para la asociación (aunque el id sigue siendo útil para CSS/JS). 

        
          <label>
            Correo
            <input name="email" type="email" autocomplete="email" required>
          </label>

⌨️ ¿Qué es <input /> y por qué es “tan grande” en HTML?

<input> crea controles interactivos para captar datos del usuario en formularios, y cambia mucho según su atributo type. Si no defines type, el valor por defecto es text.

🧪 Tipos de <input> más comunes (y cuándo usarlos)

Elegir bien el type mejora UX (teclados correctos en móvil) y validación nativa.

Ejemplo mini varios tipos:

        
          <label for="user">Usuario</label>
          <input id="user" name="user" type="text" autocomplete="username" required>

          <label for="pass">Contraseña</label>
          <input
            id="pass"
            name="pass"
            type="password"
            autocomplete="current-password"
            required
          >

          <label for="age">Edad</label>
          <input id="age" name="age" type="number" min="0" max="120">

<input> soporta un montón de atributos (min/max, minlength/maxlength, pattern, required, etc.).

🧾 Atributos clave de inputs (los “de batalla”)

name (muy importante)

Es el nombre del campo que se envía al servidor (la “llave” del dato). Esto es parte del modelo de <input> como control de formulario.

required

Marca el campo como obligatorio.

minlength / maxlength

Controlan longitud mínima/máxima del valor.

placeholder (ojo)

Es solo un hint visual, no sustituye al label (el placeholder desaparece al escribir y no debe ser la única “etiqueta” del campo).

Correcto:

        
          <label for="q">Buscar</label>
          <input id="q" name="q" type="search" placeholder="Escribe tu búsqueda">

♿ Reglas de oro de accesibilidad (labels + inputs)

  1. Cada control debe tener label asociado (explícito o implícito).
  2. No dependas solo del placeholder como label.
  3. Si por diseño no quieres mostrar el label, puedes ocultarlo visualmente pero mantenerlo en el código para tecnologías de asistencia (patrón recomendado por W3C/WAI).

Ejemplo (label oculto visualmente, pero presente):

        
          <label for="search" class="visuallyhidden">Buscar</label>
          <input id="search" name="search" type="search">
          <button type="submit">Buscar</button>

✅ Plantilla “ideal” de un campo (lista para copiar)

        
          <div class="field">
            <label for="email">Correo</label>
            <input
              id="email"
              name="email"
              type="email"
              autocomplete="email"
              required
              placeholder="tucorreo@ejemplo.com"
            >
          </div>
  • Label asociado por for/id
  • Type correcto (email) y required
  • autocomplete ayuda a UX (autorrelleno), y placeholder solo como apoyo (no reemplazo).

🧩 El control: <input type="file">

Un input con type="file" permite que el usuario seleccione uno o varios archivos de su dispositivo. Luego esos archivos se pueden subir al servidor enviando el formulario o manipularse con JavaScript usando la File API.

Ejemplo mínimo (1 archivo)

        
          <label for="cv">Sube tu CV</label>
          <input id="cv" name="cv" type="file">

Un <label> asociado mejora accesibilidad y usabilidad (el click/tap en el texto enfoca el control).

🎛️ Atributos clave para subir archivos

A) accept (filtra tipos de archivo)

accept es una sugerencia al navegador para filtrar el selector de archivos: extensiones (ej. .pdf) y/o MIME types (ej. image/jpeg). 

        
          <label for="foto">Foto de perfil (JPG/PNG)</label>
          <input id="foto" name="foto" type="file" accept="image/png, image/jpeg">
⚠️ Importante: no lo uses como validación real; el servidor debe validar el archivo.

B) multiple (seleccionar varios archivos)

Si agregas multiple, el usuario puede seleccionar más de un archivo.

        
          <label for="docs">Adjuntar documentos</label>
          <input id="docs" name="docs" type="file" multiple>

C) capture (móvil: cámara/micrófono)

Si accept indica imagen o video, capture puede sugerir usar la cámara (frontal user o trasera environment). 

        
          <label for="evidencia">Tomar foto</label>
          <input id="evidencia" name="evidencia" type="file" accept="image/*" capture="environment">

📨 Enviar archivos: enctype="multipart/form-data" + method="post"

Este es el paso que más se olvida:

  • Para subir archivos, el formulario debe enviarse como multipart/form-data.
  • Y normalmente requiere method="post".

✅ Ejemplo completo (listo para producción en HTML):

        
          <form action="/upload" method="post" enctype="multipart/form-data">
            <label for="archivo">Selecciona un archivo</label>
            <input id="archivo" name="archivo" type="file" accept=".pdf,.doc,.docx" required>
            <button type="submit">Subir</button>
          </form>

🧠 ¿Qué valor “sale” del input? (y el famoso C:\fakepath\)

  • El .value de un file input no da la ruta real del archivo; por seguridad, muchos navegadores muestran C:\fakepath\... y en general solo exponen el nombre.
  • ara trabajar con los archivos de verdad (nombre, tamaño, tipo, etc.) se usa input.files (FileList).

✅ Buenas prácticas (UX + seguridad)

✅ UX

  • Usa
  • Si aceptas múltiples tipos, sé explícito en el texto (“PDF o Word”).
  • Considera mostrar al usuario el nombre del archivo seleccionado con JS (usando files[0].name, etc.).

✅ Seguridad (no te confíes del frontend)

  • accept solo filtra el selector; no impide que alguien intente subir otra cosa. Valida del lado servidor.
  • Revisa en servidor:
    • tipo real (MIME)
    • tamaño
    • extensión
    • antivirus/escaneo si aplica (en apps serias)
    • renombrar archivos (evitar colisiones e inyecciones)
(Estas últimas son recomendaciones prácticas; la fuente citada cubre que accept no es validación y que multipart es necesario.)

✅ Validación nativa (HTML5) — sin JavaScript

HTML5 permite declarar restricciones directamente con atributos.

required (campo obligatorio)

Si un campo tiene required, el formulario no se envía si está vacío.

      
        <form>
          <label for="username">Usuario</label>
          <input id="username" name="username" type="text" required>
          <button type="submit">Enviar</button>
        </form>

Tipos semánticos (type="email", url, etc.)

Tipos semánticos (type="email", url, etc.)

  • type="email" obliga a un email válido (sintaxis).
  • type="url" obliga a una URL válida.
      
        <label for="email">Correo</label>
        <input id="email" name="email" type="email" required>

Longitud: minlength / maxlength

Sirven para restringir cuántos caracteres se aceptan.

      
        <label for="nick">Nickname (4 a 12)</label>
        <input id="nick" name="nick" minlength="4" maxlength="12" required>
      
      
        Nota útil: minlength no implica required; un campo opcional puede enviarse vacío aunque tenga minlength.

Rango numérico/fecha: min, max, step

Para números y fechas puedes validar rango y “saltos”.

      
        <label for="edad">Edad (18 a 60)</label>
        <input id="edad" name="edad" type="number" min="18" max="60" required>

pattern (regex) + title (ayuda al usuario)

pattern obliga a que el valor cumpla una expresión regular.

Además, se recomienda incluir una descripción visible y/o title para que el navegador pueda mostrarla cuando falle la validación.

Ejemplo: código postal de 5 dígitos:

      
        <label for="cp">Código postal</label>
        <input
          id="cp"
          name="cp"
          type="text"
          pattern="[0-9]{5}"
          title="Debe tener 5 dígitos (ej. 12345)"
          required
        >

♿ Validación accesible (error messages bien hechos)

La validación no solo es “bloquear envío”, también es guiar al usuario a corregir

aria-describedby para asociar mensajes al campo

aria-describedby enlaza un input con un texto descriptivo (instrucciones o error) por id. 

      
        <label for="nombre">Nombre</label>
        <input id="nombre" name="nombre" aria-describedby="nombre-error">
        <p id="nombre-error" role="alert"></p>
  • aria-describedby puede referenciar elementos por id para describir el control.
  • role="alert" ayuda a que lectores de pantalla anuncien el mensaje al aparecer (recomendación común en patrones de errores accesibles).

aria-invalid="true" para marcar campos con error

Cuando un campo falla validación, puedes marcarlo con aria-invalid="true" (pero solo después de validar, no desde el inicio).

Checklist “pro” de validación (rápido y práctico)

✅ Usa type correcto (email, url, number, etc.) para validación nativa.

✅ Usa required, minlength/maxlength, min/max/step, pattern donde aplique.

✅ Si usas pattern, agrega texto explicando el formato y/o title.

✅ Para UX/a11y: mensaje de error en texto, cerca del campo, y asociado con aria-describedby.

✅ Marca errores con aria-invalid cuando corresponda.

✅ Siempre valida también en servidor (cliente se puede manipular).

Limitaciones

1. ⚠️ Limitación #1: La validación del navegador NO sustituye la del servidor

Aunque uses required, pattern, min, etc., igual debes validar en el backend.

MDN lo dice explícitamente: la validación de restricciones en HTML no elimina la necesidad de validación del lado del servidor, porque se puede evadir de varias formas (editar HTML con DevTools, mandar un request manual, o setear valores con JavaScript sin disparar ciertas validaciones).

✅ Conclusión práctica:

  • Cliente = mejora UX
  • Servidor = seguridad y verdad

2. ⚠️ Limitación #2: Mensajes nativos del navegador (tooltips) son inconsistentes y difíciles de controlar

2.1 No puedes “estilizar” fácilmente los mensajes por defecto

Los mensajes y tooltips cambian por navegador y no se personalizan bien.

Por ejemplo, MDN explica que la validación nativa muestra mensajes “propios” del navegador y esos mensajes varían.

2.2 Accesibilidad: pueden ser vagos y no siempre anunciados

Hay documentación que advierte que los mensajes por defecto pueden ser:

  • vagos (“Please fill out this field”)
  • inconsistentes
  • y que lectores de pantalla podrían no anunciarlos de forma confiable.

✅ Solución típica:

  • desactivar nativo con novalidate
  • hacer UI de errores accesible
  • complementar con errores inline + ARIA

3. ⚠️ Limitación #3: Muchas reglas son “básicas”; validaciones complejas requieren JS

HTML5 cubre “lo típico”, pero en cuanto quieres reglas como:

  • “contraseña debe tener mayúscula + número + símbolo”
  • “si selecciona X, entonces Y es obligatorio”
  • “al menos 1 checkbox del grupo”
  • vas a necesitar JavaScript o validación servidor.

MDN: las restricciones básicas pueden declararse con atributos, pero restricciones más complejas se cubren con la Constraint Validation API.

4. ⚠️ Limitación #4: Algunos atributos NO aplican a todos los type

Este punto causa muchos bugs.

4.1 pattern no aplica a todo

pattern se usa para regex, pero aplica a ciertos tipos (p. ej. text, tel, email, url, password, search).

Por eso no funciona, por ejemplo, como “regex” para type="number".

4.2 minlength NO implica required

minlength no hace que el campo sea obligatorio; un input opcional puede enviarse vacío aunque tenga minlength.

Eso significa que si quieres:

  • mínimo 8 caracteres y obligatorio
  • debes usar: minlength="8" required.

4.3 minlength solo valida si el usuario “cambió” el valor

5. ⚠️ Limitación #5: “Bypass” voluntario con novalidate y formnovalidate

A veces es útil (guardar borrador), pero también significa que la validación puede saltarse.

  • novalidate en <form> hace que el envío evite la validación nativa.
  • formnovalidate en un botón submit puede ignorarla solo para ese envío.

✅ Implicación:

  • Si tu backend no valida, te pueden enviar cualquier cosa.

6. ♿ Limitaciones y buenas prácticas en accesibilidad de errores

6.1 aria-invalid no debe ponerse antes de validar

W3C recomienda que aria-invalid="true" se active solo después de la validación o intento de envío, no desde el inicio.

6.2 Asociar mensaje de error al campo

Para que un lector de pantalla entienda el error, lo correcto es vincular el error al control con aria-describedby. 

    
      <label for="email">Correo</label>
      <input id="email" name="email" type="email" aria-describedby="email-error">
      <p id="email-error" role="alert"></p>

6.3 role="alert": útil, pero úsalo con cuidado

MDN explica que role="alert" anuncia cambios a tecnologías de asistencia y es apropiado para mensajes importantes (como error de input), pero debe usarse con moderación por lo intrusivo.s
